## CRÍTICO — Corregir inmediatamente

### 1. Gift Card API sin autenticación
**Archivo:** `src/app/api/gift-cards/route.ts`
**Problema:** El POST que crea gift cards no requiere sesión. Cualquier persona puede crear gift cards llamando directamente a la API, sin pagar nada.
**Fix:** Agregar `auth()` check o al menos rate limiting agresivo. Idealmente integrar Stripe antes de que esto sea público real.

```ts
// Agregar al inicio del POST handler:
const session = await auth();
if (!session?.user) {
  return NextResponse.json({ error: "Unauthorized" }, { status: 401 });
}
```

### 2. Cron de reminders no está en el VPS
**Archivo:** `src/app/api/cron/reminders/route.ts`
**Problema:** El crontab del VPS no tiene el job de reminders de Caiena. Los recordatorios de citas NUNCA se envían.
**Adicionalmente:** `CRON_SECRET` no está en `.env.local`, por lo que el endpoint es **completamente público** (el if `if (secret)` no ejecuta porque secret es undefined).

**Fix:**
1. Agregar `CRON_SECRET=xxx` al `.env.local` del VPS
2. Agregar al crontab:
```
*/30 * * * * curl -s -X GET https://caienanails.com/api/cron/reminders -H "Authorization: Bearer TU_CRON_SECRET"
```

### 3. Race condition en booking por WhatsApp
**Archivo:** `src/lib/whatsapp-bot.ts:549-588`
**Problema:** La función `handleConfirm` hace check de disponibilidad y luego insert, sin transacción. Dos mensajes CONFIRM simultáneos del mismo slot pueden crear bookings duplicados.
**Fix:** Usar transacción de Drizzle con `db.transaction(async (tx) => { ... })` o un lock de aplicación.

## ALTO IMPACTO

### 4. Disco VPS al 87%
- Libre: ~11GB de 75GB
- `/var/www/caiena/public/catalog-preview/` = 48MB (25 estilos × 4 imágenes ≈ 2MB/estilo)
- Con 500+ estilos futuros: 500 × 2MB = 1GB solo en catalog-preview
- Las imágenes de upload (manos, nail-styles, services) están en 16KB ahora pero crecerán
**Acción:** Monitorear. Considerar CDN/S3 a futuro. Por ahora limpiar `.next/` builds viejos si existen.

### 5. Rate limiting solo cubre try-on
**Archivo:** `src/lib/rate-limit.ts`
**Problema:** El rate limiter existe solo para `/api/ai/try-on`. Las siguientes APIs no tienen rate limiting:
- `/api/ai/generate` — llamadas a Gemini sin límite
- `/api/ai/chat` — sin límite
- `/api/gift-cards` — (además sin auth)
- `/api/booking/request` — puede crear spam de booking requests

### 6. Rate limiter no funciona con PM2 cluster
**Archivo:** `src/lib/rate-limit.ts:2`
**Problema:** El store es en memoria. Con PM2 cluster mode (múltiples instancias), cada proceso tiene su propio store. El límite real es `MAX_REQUESTS × num_workers` por IP.
**Fix:** Mover a Redis o implementar sticky sessions en nginx para el try-on.

### 7. Sync file I/O en route async
**Archivo:** `src/app/api/admin/catalog/generate/route.ts:187`
**Problema:** `readFileSync()` y `writeFileSync()` en el handler bloquean el event loop de Node.js durante la lectura de base images (~500KB × 4) y escritura de resultados. Con PM2 cluster esto afecta a otras requests durante ese tiempo.
**Fix:** Reemplazar con `fs/promises` (`readFile`, `writeFile`).

### 8. Missing database indexes
**Archivo:** `src/db/schema.ts`
**Problema:** No hay ningún índice explícito definido. Queries frecuentes sin índice:
- `bookings.clientPhone` — historial del dashboard, loyalty count
- `bookings.date + status` — availability check
- `bookings.status` — stats, reminders
- `userProfiles.googleId` — auth callback, cada request
- `userProfiles.whatsappPhone` — phone linking check
- `whatsappSessions.phone` — cada mensaje WA
- `nailStyles.published` — catalog query

**Fix:**
```ts
// En schema.ts
import { index } from "drizzle-orm/pg-core";

export const bookings = pgTable("bookings", { ... }, (t) => [
  index("bookings_client_phone_idx").on(t.clientPhone),
  index("bookings_date_status_idx").on(t.date, t.status),
]);

export const userProfiles = pgTable("user_profiles", { ... }, (t) => [
  index("user_profiles_google_id_idx").on(t.googleId),
  index("user_profiles_wa_phone_idx").on(t.whatsappPhone),
]);
```

### 11. Sin limpieza de sesiones expiradas
`whatsappSessions` y `whatsappLinkTokens` nunca se limpian automáticamente. Solo se eliminan cuando el mismo usuario vuelve. Con el tiempo esto acumulará filas muertas.
**Fix simple:** Agregar a la cron de reminders: `await db.delete(whatsappSessions).where(lt(whatsappSessions.expiresAt, new Date()))`

### 12. Gift cards sin pago real
`src/app/api/gift-cards/route.ts` crea la gift card sin procesar ningún pago. Es un placeholder funcional. Antes de promover gift cards, integrar Stripe payment intent.

### 13. Entropy del booking token bajo
`src/app/api/booking/request/route.ts:21` — El token es 10 hex chars (5 bytes = 40 bits). Mejorar a 16 bytes (32 hex chars) para mayor seguridad.

### 14. WhatsApp bot: staff cancel no notifica
`src/lib/whatsapp-bot.ts:648` — Cuando el staff cancela, `otherPhone = party === "staff" ? booking.clientPhone : null`. Para el caso `party === "client"`, no se notifica al staff. Hay un `// TODO: staff phone lookup` en el código.

### 15. `SESSION_SECRET` vs `AUTH_SECRET` en .env
El VPS tiene `SESSION_SECRET` en las env vars pero NextAuth v5 usa `AUTH_SECRET`. `SESSION_SECRET` podría ser un env var huérfano o de otra app. Verificar si es usado.

## BAJO — Mejoras futuras

### 16. Metadata de páginas en inglés cuando locale=EN
Los `metadata.title` y `metadata.description` de Next.js están hardcodeados en español. No afectan al usuario directo pero sí a SEO cuando Google indexa con locale EN.

### 17. Paginación en historial de bookings
`src/components/user-dashboard.tsx:80` — `.limit(20)` en historial. Clientes con muchas visitas solo ven las últimas 20. No hay "ver más".

### 18. Stats route hace 5 queries separadas
`src/app/api/admin/stats/route.ts` — 5 queries secuenciales sin Promise.all. Pueden paralelizarse.

### 19. Catalog generate: timeout sin manejo
El route de generación puede tomar 40+ segundos (4 bases × 10s Gemini + retries). Next.js tiene límite de 30s por defecto en edge functions, y nginx tiene 300s de timeout configurado. En cluster PM2 esto amarra un worker por minuto entero. Considerar mover a background job.

### 20. No hay OG images dinámicas
Las páginas no tienen `generateMetadata` con `openGraph.images`. Compartir links en redes sociales no muestra preview atractivo.

## iOS App Readiness

### Bloqueadores actuales
1. **Sin auth móvil**: NextAuth v5 usa cookies. iOS URLSession puede manejar cookies pero es complejo y poco estándar para apps nativas.
2. **Sin Bearer token endpoint**: Las apps iOS/Android esperan `Authorization: Bearer <token>` para llamadas a API.
3. **Sin push notification backend**: No hay APNs configuration ni endpoint para registrar device tokens.
4. **Google Sign-In en iOS**: Requiere configurar el mismo OAuth client con `com.googleusercontent.apps.CLIENT_ID://` redirect URI.

### Endpoints existentes que puede consumir iOS directamente
```
GET  /api/services          — lista de servicios (público) ✅
GET  /api/catalog           — catálogo de estilos (público) ✅
GET  /api/availability      — slots disponibles (público) ✅
POST /api/ai/try-on         — try-on con Gemini ✅
GET  /api/user/me           — perfil del usuario (requiere auth)
GET  /api/bookings          — citas del usuario (requiere auth)
POST /api/booking/request   — crear booking request (requiere auth)
```

### Nuevos endpoints necesarios para iOS
```
POST /api/auth/mobile/google   — recibe Google ID token, retorna JWT propio
GET  /api/auth/mobile/refresh  — refresh token
POST /api/auth/mobile/device-token — registrar APNs device token
GET  /api/user/loyalty         — loyalty data para el cliente
POST /api/bookings/[id]/cancel — cancelar cita desde app
```

### Arquitectura sugerida para auth móvil
```
iOS App → GoogleSignIn SDK → Google ID Token
       → POST /api/auth/mobile/google { idToken }
       → Backend verifica con Google tokeninfo API
       → Crea/upsert userProfile
       → Retorna { accessToken (JWT, 1h), refreshToken (30d) }
       → iOS guarda en Keychain
       → Todas las requests: Authorization: Bearer <accessToken>
```
